## UI Component Specification

### Monitoring Section Structure

Add a new `<section>` to SettingsPanel.tsx following existing patterns:

```tsx
<section style={{ marginBottom: '24px' }}>
  <h2>监控控制</h2>
  {/* Status display */}
  <div style={{ marginBottom: '16px' }}>
    <strong>当前状态：</strong>
    <span style={{ color: {statusColor} }}>{statusText}</span>
  </div>

  {/* Visible region warning */}
  <div style={{ marginBottom: '16px', color: '#856404' }}>
    ⚠️ MSS 模式仅捕获屏幕可见区域。请确保监控区域不被遮挡。
  </div>

  {/* FPS config (read-only for Phase 2, editable later) */}
  <div style={{ marginBottom: '16px' }}>
    <strong>捕获帧率：</strong> {fps} FPS
  </div>

  {/* Start/Stop buttons */}
  <div>
    <button
      onClick={handleStartMonitoring}
      disabled={isMonitoring || isStarting || isStopping}
      style={{ marginRight: '10px', padding: '8px 16px' }}
    >
      开始监控
    </button>
    <button
      onClick={handleStopMonitoring}
      disabled={!isMonitoring || isStarting || isStopping}
      style={{ padding: '8px 16px' }}
    >
      停止监控
    </button>
  </div>

  {/* Error message */}
  {error && (
    <div style={{ marginTop: '16px', padding: '10px', backgroundColor: '#f8d7da', borderRadius: '4px', color: '#721c24' }}>
      监控失败：{error}。请检查配置后重试。
    </div>
  )}
</section>
```

### States and Transitions

| State | Status Text | Status Color | Start Button | Stop Button | Error Display |
|-------|-------------|--------------|--------------|-------------|---------------|
| Idle | 未启动 | #0f0f0f (default) | enabled | disabled | hidden |
| Starting | 启动中... | #fd7e14 (orange) | disabled | disabled | hidden |
| Running | 运行中 | #28a745 (green) | disabled | enabled | hidden |
| Stopping | 停止中... | #fd7e14 (orange) | disabled | disabled | hidden |
| Stopped | 已停止 | #0f0f0f (default) | enabled | disabled | hidden |
| Error | 错误 | #dc3545 (red) | enabled | disabled | visible |

**Button states:**
- Disabled: `opacity: 0.6`, `cursor: not-allowed` (CSS classes from App.css)
- Enabled: Normal cursor, full opacity
- Hover: Border color changes to `#396cd8` (from App.css)
- Active: Background color changes to `#e8e8e8` (from App.css)

### Accessibility Requirements

1. **Keyboard navigation:** All buttons must be keyboard accessible (Tab to focus, Enter/Space to activate)
2. **Screen reader support:** Status changes should be announced via `aria-live="polite"` region
3. **Color contrast:** All text must meet WCAG AA contrast ratios:
   - Status green (#28a745) on white (#ffffff): 4.5:1 ✓
   - Status orange (#fd7e14) on white (#ffffff): 2.6:1 ✗ → Use darker orange (#e67e22) for WCAG AA compliance
   - Status red (#dc3545) on white (#ffffff): 4.5:1 ✓
   - Warning text (#856404) on white (#ffffff): 4.5:1 ✓
4. **Button focus:** Visible focus indicator (outline or border color change) when keyboard navigated

### Responsive Constraints

Phase 2 UI is contained within existing SettingsPanel. No responsive design changes required. The existing layout uses fixed padding and margins that work on typical desktop window sizes (800px+ width minimum, as this is a Windows desktop app).

## State Management Requirements

The UI component must track the following local state:

```tsx
interface MonitoringState {
  status: 'idle' | 'starting' | 'running' | 'stopping' | 'stopped' | 'error';
  error: string | null;
  fps: number; // Read from config, e.g., 5-10 FPS default
  isMonitoring: boolean; // Derived from status
  isStarting: boolean; // Derived from status
  isStopping: boolean; // Derived from status
}
```

**Tauri event listeners:**
- Listen for `monitoring-status` events to update status from backend
- Listen for `monitoring-error` events to display error messages
- Use `unlisten()` cleanup in `useEffect` to avoid memory leaks

## Implementation Notes

1. **Follow existing patterns:** Use the same inline style approach as the existing SettingsPanel. No new CSS classes or external libraries.
2. **Reuse App.css styles:** Buttons automatically inherit hover/active states from App.css. No custom button styling needed.
3. **Message display:** Use the same pattern as the existing message box: light gray background, rounded corners, margin top.
4. **Section structure:** Follow the existing pattern: `<section>` with `<h2>` heading and inline styles for spacing.
5. **Error handling:** Match the existing error handling pattern: try-catch with `getErrorMessage()` helper, display to user.
6. **Status colors:** Update the orange color to `#e67e22` for WCAG AA compliance when implementing.
7. **Config freeze:** When starting monitoring, use the existing `create_frozen_config()` command to lock the config. Live edits should not affect active monitoring (per CONTEXT.md D-36).
8. **Status persistence:** The status should reflect the backend monitoring state, not just local UI state. Use Tauri events to stay in sync.
